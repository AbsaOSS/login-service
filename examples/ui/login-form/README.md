# Login Form UI Example

A minimal browser-based example that authenticates a user via **Microsoft Entra ID (Azure AD)**
using [MSAL.js](https://github.com/AzureAD/microsoft-authentication-library-for-js) and
exchanges the resulting Entra access token for a **Login Service JWT**.

## Authentication flow

```mermaid
sequenceDiagram
    actor Browser
    participant Entra as MS Entra (Azure AD)
    participant LS as Login Service

    Browser->>Entra: click "Login using Entra"<br/>(MSAL popup / redirect)
    Entra-->>Browser: Entra access token

    Browser->>LS: POST /token/generate<br/>Authorization: Bearer &lt;entra-token&gt;
    LS-->>Browser: { token: "...", refresh: "..." }

    Note over Browser: display LS tokens
```

## Project structure

```
login-form/
├── src/                        # JavaScript source modules
│   ├── app.js                  # Entry point — wires modules together, registers event listeners
│   ├── auth.js                 # MSAL authentication logic (login, token acquisition, logout)
│   ├── login-service-client.js # HTTP client for the Login Service token-exchange endpoint
│   └── ui.js                   # DOM rendering helpers (status messages, token display, etc.)
├── dist/                       # Compiled output (generated by `npm run build`, git-ignored)
│   └── bundle.js               # Single self-contained browser bundle
├── config.js                   # Runtime configuration (fill in before running)
├── index.html                  # Single-page UI — loads config.js then dist/bundle.js
├── server.js                   # Express server — serves the app on http://localhost:3000
└── package.json                # npm metadata, build and start scripts
```

`src/` follows a strict separation of concerns:

| Module | Responsibility |
|--------|----------------|
| `auth.js` | All MSAL / Entra interactions — no DOM, no HTTP |
| `login-service-client.js` | Login Service HTTP calls — no DOM, no MSAL |
| `ui.js` | All DOM manipulation — no auth logic, no HTTP |
| `app.js` | Orchestration only — imports the three modules above and wires event listeners |

## Prerequisites

| Requirement | Notes |
|-------------|-------|
| Node.js | v18 or later |
| Running Login Service | Default: `http://localhost:9090`. Configure `entra` in the server's `application.yaml` (`order > 0`). |
| Entra App Registration | Must be a **Single-Page Application (SPA)** type with the page URL added as a redirect URI. |
| CORS on Login Service | The Login Service must allow `Access-Control-Allow-Origin` for the origin serving this page. |

## Configuration

1. Open [`config.js`](config.js) and fill in your values:

   ```js
   const LOGIN_FORM_CONFIG = {
     loginServiceHost: "http://localhost:9090",   // ← login-service.example.host
     entra: {
       tenantId: "your-tenant-id",               // ← login-service.example.entra.tenant-id
       clientId: "your-client-id",               // ← login-service.example.entra.client-id
       scopes:   ["api://your-client-id/.default"],
     },
   };
   ```

   The same values are documented (commented out) in
   [`../src/main/resources/example.application.yaml`](../src/main/resources/example.application.yaml)
   under `login-service.example.entra`.

2. Make sure the Login Service is configured with the `entra` provider and the token's audience
   matches `login-service.rest.auth.provider.entra.audiences`.

## Running

`npm install` installs dependencies and runs the build step automatically via the `prepare`
lifecycle hook.

```bash
cd examples/ui/login-form
npm install   # installs dependencies and builds dist/bundle.js
npm start     # starts the Express server on http://localhost:3000
```

Then open <http://localhost:3000> in a browser and click **Login using Microsoft Entra**.

> **Note:** Opening `index.html` as a `file://` URL will not work due to MSAL redirect-URI restrictions.

### Rebuilding after source changes

If you edit any file under `src/`, regenerate the bundle before refreshing the browser:

```bash
npm run build
```

## How the build works

[esbuild](https://esbuild.github.io/) bundles `src/app.js` (and all its imports, including
`@azure/msal-browser`) into a single minified IIFE at `dist/bundle.js`. This means:

- No CDN dependency — MSAL is resolved from `node_modules` at build time.
- `index.html` only needs two script tags: `config.js` (runtime config) and `dist/bundle.js`.
- `dist/` is git-ignored; the bundle is always regenerated locally via `npm install`.

## Output

After a successful login the page displays:

* **Login Service Access Token** – the short-lived JWT to use in downstream service calls.
* **Login Service Refresh Token** – the long-lived JWT for obtaining new access tokens.

Both tokens can be copied to the clipboard with the **Copy** button.
